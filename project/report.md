## 2. Постановка задачи и контекст

Опишите:

1. **Предметную область и задачу:**
   - Анализ рынка недвижимости и автоматизация машинного обучения.
   - Сервис решает задачу регрессии: прогноз рыночной стоимости жилых объектов (SalePrice) на основе набора признаков (площадь, состояние, год постройки, локация и т.д.). Помимо предсказательной части, сервис выполняет задачу автоматизированного контроля качества данных, проверяя входящие датасетов до того, как они попадут в ML-модель. Потенциальными пользователями являются Data Scientists и аналитики данных, работающие в компаниях по оценке недвижимости или риелторских агентствах.

2. **Формулировку задачи в терминах ML/ИИ:**
   - Входные данные:
      - Для обучения: Табличные данные (CSV/DataFrame), содержащие характеристики объектов недвижимости (площадь, тип жилья, год постройки, локация, качество материалов и прочее). Данные включают как количественные признаки, так и категориальные.
      - Для инференса: Аналогичный набор признаков для новых объектов недвижимости, для которых необходимо получить оценку стоимости.
   - Целевая переменная / Выход модели:
      - Целевая переменная: SalePrice (цена продажи объекта).
      - Выход модели: Непрерывное числовое значение, представляющее предсказанную стоимость недвижимости. В текущей реализации модель предсказывает логарифм цены (log(SalePrice)), что помогает стабилизировать дисперсию и улучшить качество предсказаний, который затем конвертируется в реальную стоимость через обратное преобразование (expm1).

3. **Целевые метрики качества:**
   - RMSE (Root Mean Squared Error):
      - Описание: Корень из среднего квадрата отклонений предсказаний от реальных значений.
      - Почему выбрана: Это стандартная метрика для регрессии, которая сильно штрафует модель за крупные ошибки. В сфере недвижимости критически важно избегать значительных отклонений от рыночной цены, поэтому RMSE позволяет сфокусироваться на точности наиболее «трудных» случаев.

   - MAPE (Mean Absolute Percentage Error):
      - Описание: Средняя абсолютная процентная ошибка, показывающая, на сколько процентов в среднем ошибается модель.
      - Почему выбрана: Эта метрика обладает высокой интерпретируемостью для бизнеса. Клиенту или риелтору понятнее услышать: «Модель ошибается в среднем на 5-8% от стоимости объекта», чем оперировать абстрактными единицами RMSE.

   - R² (Коэффициент детерминации):
      - Описание: Показывает долю дисперсии целевой переменной, объясненную моделью.
      - Почему выбрана: Помогает оценить «общую полезность» модели. R² близкий к 1 означает, что модель эффективно улавливает основные рыночные факторы, влияющие на цену.

## 4. Модели и подходы

`03_Modeling_and_Evaluation.ipynb`
1. **Базовые (baseline) модели:**
   - В качестве отправной точки для оценки качества прогнозов был использован DummyRegressor.
   - Подход: Модель предсказывала константное среднее значение целевой переменной (SalePrice) для всех объектов.
   - Зачем это нужно: DummyRegressor задает «наивный» порог точности. Любая серьезная ML-модель должна значительно превосходить этот результат, чтобы доказать свою прикладную ценность.
   - Результаты: RMSE: 0.3996, MAE: 0.3102, r2: -0.0007

2. **Улучшенные модели и эксперименты:**
   - Поверх DummyRegressor были внедрены следющие алгоритмы, способные находить зависимости:
   - Линейный базис: Lasso Regression была выбрана как инструмент, который автоматически выполняет отбор признаков, обнуляя коэффициенты наименее значимых переменных.
   - Древовидные модели: Random Forest и HistGradientBoostingRegressor были добавлены для учета эффектов взаимодействия признаков.
   - Финальный ансамбль: StackingRegressor позволил объединить сильные стороны линейных и древовидных алгоритмов. Мета-моделью выступила Ridge, которая эффективно усредняет предсказания, уменьшая дисперсию итоговой ошибки.
   *Оптимизация гиперпараметров: методы регуляризации*
   - Random Forest
      - max_depth & max_leaf_nodes: позволяет модели быть достаточно гибкой, но не «зазубривать» обучающую выборку.
      - ccp_alpha: эксперименты со значениями направлены на обрезаниче ветвей дерева по заданному погору, что критически важно для работы с зашумленными данными.
      - max_features: обеспечивают разнообразие деревьев в ансамбле, что снижает общую дисперсию ошибки модели.
   - HistGradientBoosting
      - l2_regularization: напрямую штрафует веса модели, ограничивая влияние аномальных выбросов на предсказания.
      - learning_rate: подбор темпа обучения позволил найти баланс между скоростью сходимости и качеством обобщения.
      - max_leaf_nodes: жесткое ограничение количества листьев значительно снижает дисперсию каждого отдельного дерева. Это превращает каждое дерево в «слабую модель», что заставляет ансамбль обучаться на обобщенных закономерностях и эффективно предотвращает переобучение.

- Проведенные эксперименты
   `02_Data Cleaning_Feature Engineering.ipynb`
      - Feature Engineering: Создание агрегированных метрик (TotalSF, TotalBath) и HouseAge дало наибольший прирост точности, подтвердив, что качество признаков важнее сложности архитектуры.

      - Трансформация таргета: Переход к log(SalePrice) устранил гетероскедастичность и привел распределение остатков к нормальному виду, что критично для работы линейных моделей.

      - Обработка категорий: для признаков с выраженным ранговым смыслом применен Ordinal Encoding, что позволяет сохранить логическую упорядоченность данных. Для номинативных категорий использовано One-Hot Encoding, что предотвращает ложное создание порядковых зависимостей и обеспечивает корректное представление признаков в модели.

      - Борьба с мультиколлинеарностью: Анализ VIF (Variance Inflation Factor) позволил выявить и исключить избыточные признаки, повысив интерпретируемость коэффициентов Lasso.
   
   `03_Modeling_and_Evaluation.ipynb`
      - Архитектура Stacking: Объединение разнородных алгоритмов (Lasso, Random Forest, HistGradientBoosting) позволило компенсировать индивидуальные недостатки моделей: линейная регрессия стабилизирует предсказания, а древовидные ансамбли улавливают сложные нелинейные зависимости.

      - Оптимизация гиперпараметров: С помощью GridSearchCV был проведен подбор гиперпараметров модели (например, таких как alpha для Lasso и max_depth для леса). Это позволило найти баланс между смещением и дисперсией, минимизировав риск переобучения.

      - Финальный пайплайн: Весь процесс — от заполнения пропусков до мета-прогноза — был упакован в единый объект Pipeline. Это гарантирует отсутствие «утечки данных» при инференсе и позволяет разворачивать модель в один клик.

3. **Выбор финальной модели:**
   - В качестве итогового решения выбран Stacking Regressor. Несмотря на то, что прирост метрики R^2 по сравнению с Lasso составил менее 1%, выбор ансамбля обоснован его качественными характеристиками.
      - Повышенная стабильность: В отличие от одиночной Lasso-регрессии, которая сильно зависит от линейности связей и чувствительна к мультиколлинеарности, Stacking усредняет подходы разных семейств алгоритмов (линейные и древовидные). Это делает итоговый прогноз более устойчивым к специфическим выбросам в данных.
      - Уверенность в прогнозе: За счет использования мета-модели Ridge, ансамбль сглаживает экстремальные предсказания базовых моделей, что снижает риск получения неадекватной цены для нестандартных объектов недвижимости.
      - Trade-off сложности и надежности: Хотя Stacking обучается дольше и имеет более сложную архитектуру, в задаче оценки недвижимости приоритет отдается надежности прогноза. Дополнительные затраты времени на обучение (исчисляемые секундами) полностью оправданы минимизацией дисперсии ошибки.

      - Информационный (INFO):
         - Старт сервиса и статус загрузки ML-модели.
         - Метаданные входящих запросов (имя файла, размер таблицы, количество строк).
         - Метрики производительности (latency_ms) — время, затраченное на предсказание или EDA-анализ.  
      - Предупреждения (WARNING):
         - Попытки загрузки файлов некорректного формата (не .csv).
         - Запросы к эндпоинтам при не полностью инициализированной модели.
      - Критические ошибки (ERROR / CRITICAL):
         - Ошибки чтения файлов или сбои внутри ML-пайплайна.
         - Проблемы с путями к файлам моделей или отсутствующие зависимости.
   - Все логи сохраняются в файл logs/api.log с указанием точного времени и модуля, что позволяет быстро восстановить хронологию событий при сбое.
   - Понять, что сервис работает корректно, можно двумя способами:
      - Автоматическая проверка (Health Check): Эндпоинт GET /health возвращает статус ok и флаг model_ready: true. Это основной индикатор того, что и сервер, и ML-модель готовы к нагрузке.
      - Инспекция логов: Отсутствие записей уровня ERROR в api.log и наличие подтверждений «✅ Модель успешно загружена» при старте гарантируют штатную работу системы.
   - Интерактивная документация: Через встроенный интерфейс Swagger UI (/docs) можно вручную протестировать каждый эндпоинт и мгновенно увидеть код ответа (200 OK).

2. **Конфигурации:**
   - Настройки API:
      - API_PORT: Порт, на котором поднимается FastAPI (по умолчанию 8000). Позволяет избежать конфликтов, если порт занят другим сервисом.
      - API_RELOAD: Режим автоматической перезагрузки сервера при изменении кода.
   - Логирование:
      - LOG_LEVEL: Уровень детализации логов (INFO, WARNINGS, ERROR).
      - CONFIG_NAME: Имя основного конфигурационного файла, что позволяет быстро переключаться между разными версиями настроек проекта.
   - base.yaml (Общая структура):
      - Пути (dirs): Относительные пути к папкам артефактов, данных, логов и моделей. Это позволяет коду работать корректно вне зависимости от того, в какой папке на диске лежит проект.
      - Файлы (files): Имена исходных файлов данных (train.csv, test.csv).
      - Общие параметры (general): Глобальные константы, такие как random_state для воспроизводимости экспериментов и имя целевой переменной (target).
      - Подпапки артефактов: Четкая структура для хранения EDA-отчетов, графиков и промежуточных данных.
   - data_params.yaml:
      - Специфические настройки предобработки данных (например, списки признаков, типы колонок или параметры нормализации).
   - best_meta_model.json:
      - Параметры лучшей обученной модели, полученные в результате оптимизации гиперпараметров. Используется скриптами обучения и инференса для автоматической сборки финального пайплайна.

   - Благодаря выносу настроек в .env и configs/, без изменения кода можно перенастроить:
      - Инфраструктуру: Порт API, режим авто-перезагрузки (reload) и уровень логирования (DEBUG/INFO).
      - Пути: Расположение папок с данными, моделями и логами (позволяет переносить проект между дисками/серверами).
      - Данные: Имена входных CSV-файлов и название целевой переменной (target).
      - Эксперименты: Параметры воспроизводимости (random_state) и гиперпараметры модели в JSON-конфиге.
      - Профили: Быстрое переключение между разными сценариями работы через смену CONFIG_NAME.

3. **Безопасность:**
   - Предотвращение утечки секретов: Для исключения попадания чувствительных данных в систему контроля версий (Git) используется файл .gitignore. В него добавлены исключения для:
      - Файла переменных окружения .env (содержит локальные настройки и доступы) и локальны
      - Служебных папок Python (__pycache__, .venv).

   - Использование шаблонов:
      - Вместо реальных секретов в репозиторий закоммичен файл .env.example. Он служит инструкцией для других разработчиков: какие переменные нужно создать локально, не раскрывая при этом реальные параметры среды.

   - Анализ чувствительных данных:
      - В репозитории отсутствуют токены, пароли, ключи доступа к API или персональные данные пользователей. Все данные в папке data/raw/ являются обезличенными тренировочными выборками для обучения модели.

## 8. Ограничения и дальнейшая работа

- Формат входных данных: Сервис жестко завязан на формат .csv. При попытке загрузить .xlsx, .json или .parquet напрямую через эндпоинт /predict/csv система выдаст ошибку. Это ограничивает интеграцию с системами, использующими другие стандарты обмена данными.

- Зависимость от структуры: Несмотря на наличие предобработки, модель ожидает строго определенный набор признаков. Если во входном файле будут отсутствовать колонки, которые кастомный трансформер CombinedFeaturesAdded использует для генерации новых фич, инференс упадет с ошибкой.

- Отсутствие авторизации: API открыто для всех в локальной сети (0.0.0.0). В текущем виде любой пользователь сети может завалить сервер бесконечными запросами, так как нет ограничений по количеству запросов.

- что бы было реализовано, если бы было больше времени:
  - Docker: Контейнеризация проекта для запуска одной командой на любой ОС без установки библиотек.
  - Расширение интерфейсов: Добавление поддержки JSON-запросов (для предсказания по одному объекту из веб-формы) и поддержка других форматов файлов (Excel/Parquet).
  - Векторизованная валидация данных на уровне DataFrame: Написать детальную схему для каждой строки CSV, чтобы API не просто падало с ошибкой, а четко говорило: «В строке 45 в колонке 'YearBuilt' пришла строка вместо числа».
  - MLFlow: Внедрение графического интерфейса для централизованного хранения и анализа артефактов (метрик, графиков и весов моделей).
  - DVC: Для контроля версий данных. Если через месяц придут новые данные по домам, DVC позволит легко переключаться между старой и новой версией датасета без бардака в папках.
