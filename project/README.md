- **Краткое описание (2-4 предложения):**  

Проект представляет собой автоматизированный ML-конвейер для прогнозирования стоимости недвижимости на основе датасета Ames Housing. В работе используются методы ансамблевого обучения (StackingRegressor с Lasso, Random Forest и HistGradientBoosting) и конвейеры scikit-learn для обеспечения воспроизводимости данных. Результат — это CLI-утилита и REST API, которые позволяют выполнять обучение моделей, автоматическую генерацию EDA-отчётов и инференс с поддержкой логирования всех этапов жизненного цикла модели.

## 2. Структура проекта

Проект организован в следующей структуре:

- `reports` —  отчеты по результатам ноутбуков.
- `report.md` – отчёт по проекту (постановка задачи, данные, эксперименты, результаты).
- `self-checklist.md` – чеклист самопроверки проекта перед сдачей.
- `notebooks/` – экспериментальные ноутбуки:
  - EDA, прототипы моделей, отдельные эксперименты.
- `src/` – основной код проекта:
  - api.py — реализация REST API сервиса (FastAPI) для получения предсказаний в реальном времени.
  - cli.py — интерфейс командной строки для управления жизненным циклом модели (обучение, оценка, инференс).
  - pipeline.py — сборка и конфигурация Scikit-learn пайплайнов для автоматизации обработки данных.
  - transformers.py — кастомные классы трансформации данных (наследуются от BaseEstimator и TransformerMixin).
  - core.py — базовые аналитические функции, расчет метрик и инструменты интерпретации моделей.
  - config.py — централизованное управление путями, переменными окружения (.env) и настройками проекта.
  - logger_setup.py — конфигурация системы логирования для отслеживания процесса обучения и работы сервиса.
  - viz.py — функции для построения графиков анализа признаков (EDA) и оценки результатов моделирования.

- `data/` – демонстрационные/учебные данные:
- `configs/` – конфигурационные файлы:
  - `.yaml`, `.json`, `.env.example` 
- `tests/` – тесты (юнит-тесты, простые проверки).
- `logs/` — системные журналы. Здесь хранятся файлы логов (например, api.log). 
- `artifacts/` – сохранённые модели, отчёты, результаты экспериментов.
- `pyproject.toml` / `uv.lock` — конфигурация окружения и жесткая фиксация версий библиотек.
- `.env.example` — шаблон переменных окружения для быстрой настройки сервиса.
- `models/` — cохранение обученного пайплайна для последующего использования в CLI или API.

## 4. Как запустить проект

### 4.1. Запуск обучения модели

```bash
cd project
source .venv/bin/activate      # при необходимости
uv run python -m src.cli train data/raw/train.csv    
```

### 4.2. Запуск сервиса (API/веб-интерфейс)

```bash
cd project
source .venv/bin/activate      # при необходимости
uv run python -m src.api         # пример: FastAPI/Flask сервис
```

- Порт по умолчанию: 8000 (настраивается в src/config.py через API_PORT).
- Системные эндпоинты 
  - GET /health
    - Назначение: Проверка работоспособности сервиса.
    - Возвращает: Статус системы и подтверждение загрузки ML-модели в память. Если модель не загружена, возвращает ошибку 503 Service Unavailable.
    - POST /predict/csv
    - Назначение: Пакетное предсказание цен на недвижимость из файла.
    - Входные параметры: Файл формата .csv. Файл должен содержать признаки, использованные при обучении. Поле Id является опциональным.
  - POST /quality
    - Назначение: Быстрая проверка "пригодности" данных на основе агрегированных метрик.
    - Принимает (JSON): * n_rows, n_cols, max_missing_share, numeric_cols, categorical_cols.
    - Возвращает: Оценку качества quality_score (0–1) и флаги проблем (например, too_few_rows).
  - POST /quality-from-csv
    -  Назначение: Полный автоматизированный анализ качества данных из CSV.
    - Входные параметры: Файл .csv.
    - Логика: Вызывает ядро EDA (summarize_dataset, missing_table), рассчитывает флаги качества и возвращает вердикт — можно ли использовать этот датасет для обучения модели.
   - Lifespan: Загрузка модели final_production_model.joblib происходит один раз при старте сервера, что исключает задержки при каждом запросе.
   - Logging: Все действия (загрузка, ошибки, время обработки latency_ms) фиксируются в logs/api.log.
   - Валидация: Схемы данных строго типизированы с помощью Pydantic, что предотвращает обработку некорректных запросов.
   
  - Тестирование работоспособности
    - открыть браузер по адресу: http://127.0.0.1:8000/docs
    - Там можно нажать кнопку "Try it out", ввести данные и сразу увидеть ответ сервера.

## 8. Ограничения и дальнейшая работа

- Формат входных данных: Сервис жестко завязан на формат .csv. При попытке загрузить .xlsx, .json или .parquet напрямую через эндпоинт /predict/csv система выдаст ошибку. Это ограничивает интеграцию с системами, использующими другие стандарты обмена данными.

- Зависимость от структуры: Несмотря на наличие предобработки, модель ожидает строго определенный набор признаков. Если во входном файле будут отсутствовать колонки, которые кастомный трансформер CombinedFeaturesAdded использует для генерации новых фич, инференс упадет с ошибкой.

- Отсутствие авторизации: API открыто для всех в локальной сети (0.0.0.0). В текущем виде любой пользователь сети может завалить сервер бесконечными запросами, так как нет ограничений по количеству запросов.

- что бы было реализовано, если бы было больше времени:
  - Docker: Контейнеризация проекта для запуска одной командой на любой ОС без установки библиотек.
  - Расширение интерфейсов: Добавление поддержки JSON-запросов (для предсказания по одному объекту из веб-формы) и поддержка других форматов файлов (Excel/Parquet).
  - Векторизованная валидация данных на уровне DataFrame: Написать детальную схему для каждой строки CSV, чтобы API не просто падало с ошибкой, а четко говорило: «В строке 45 в колонке 'YearBuilt' пришла строка вместо числа».

  - MLflow: Чтобы не записывать результаты тестов в блокнот, а автоматически видеть все графики и метрики стэкинга в красивом интерфейсе.
  - DVC: Для контроля версий данных. Если через месяц придут новые данные по домам, DVC позволит легко переключаться между старой и новой версией датасета без бардака в папках.
